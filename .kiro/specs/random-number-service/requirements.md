# Requirements Document

## Introduction

A two-tier web application consisting of a Bootstrap-based frontend and a FastAPI-based backend. The backend exposes a random number generation service via a REST API. The frontend displays the generated random number and provides a refresh mechanism to request a new number from the service.

## Glossary

- **Random_Number_Service**: The FastAPI-based backend application that generates and returns random numbers via a REST API.
- **Frontend**: The Bootstrap-based web interface that displays random numbers and provides user interaction controls.
- **Random_Number**: An integer generated by the Random_Number_Service on each request.
- **Refresh_Button**: A UI control on the Frontend that triggers a new request to the Random_Number_Service.
- **GET_Endpoint**: The HTTP GET route exposed by the Random_Number_Service that returns a randomly generated number.
- **Docker_Compose**: The container orchestration tool used to define and run the multi-container local development environment.
- **Application_Container**: The Docker container running the Random_Number_Service and Frontend locally via Docker_Compose.
- **ECS_Cluster**: The Amazon ECS cluster in the us-east-2 region used for cloud deployment of the application.
- **Env_File**: The .env file containing access tokens and configuration variables required for authentication and deployment.
- **Requirements_File**: The requirements.txt file at the project root listing all Python package dependencies needed by the Random_Number_Service.
- **Virtual_Environment**: The .venv directory at the project root containing the pre-created Python virtual environment used for local development.
- **Infra_Directory**: The `infra` directory at the project root containing all Terraform infrastructure code for provisioning and managing cloud resources.
- **VPC**: The AWS Virtual Private Cloud created by the infrastructure code to isolate and host all networking resources for the application.
- **Application_Load_Balancer**: The AWS Application Load Balancer that provides a public-facing HTTP endpoint for routing traffic to the ECS_Cluster services.
- **Public_Subnet**: A subnet within the VPC that has a route to the Internet_Gateway, enabling public internet access.
- **Internet_Gateway**: The AWS Internet Gateway attached to the VPC to allow inbound and outbound internet traffic.
- **Security_Group**: The AWS Security Group that controls inbound and outbound network traffic rules for the Application_Load_Balancer and ECS tasks.
- **Route_Table**: The AWS Route Table associated with subnets in the VPC that defines routing rules for network traffic.
- **Terraform**: The infrastructure-as-code tool used to define, provision, and manage all cloud infrastructure resources declaratively.
- **Terraform_State**: The state file managed by Terraform that tracks the current state of provisioned infrastructure resources.
- **GitHub_Repository**: The remote Git repository on GitHub used for version control and as the source for CI/CD pipelines.
- **GitHub_Actions**: The CI/CD platform integrated with GitHub used to automate build and deploy pipelines.
- **Build_Pipeline**: The GitHub_Actions workflow that builds Docker container images for the Frontend and Random_Number_Service.
- **Deploy_Pipeline**: The GitHub_Actions workflow that deploys built container images to the ECS_Cluster and verifies deployment health.
- **ECS_Task_Health_Check**: The post-deployment verification step that confirms ECS tasks reach and maintain a healthy running state after deployment.
- **Access_Token**: The GitHub personal access token stored in the Env_File, used for authenticating GitHub API operations such as repository creation.
- **ECR**: Amazon Elastic Container Registry, the AWS-managed Docker container image registry used to store and distribute container images for the Frontend and Random_Number_Service.
- **Nginx_Container**: The Docker container running Nginx that serves the static Frontend assets and is deployed as an ECS task.
- **Fargate**: The AWS Fargate serverless compute engine for ECS that runs containers without requiring management of EC2 instances.
- **CloudWatch_Logs**: The Amazon CloudWatch Logs service used to collect, store, and view log output from ECS tasks.
- **ALB_Routing_Rule**: A listener rule on the Application_Load_Balancer that routes incoming requests to specific target groups based on URL path patterns.

## Requirements

### Requirement 1: Random Number Generation API

**User Story:** As a client application, I want to call a GET endpoint on the Random_Number_Service, so that I can receive a randomly generated number.

#### Acceptance Criteria

1. WHEN a GET request is made to the GET_Endpoint at the `/api` path prefix, THE Random_Number_Service SHALL return a JSON response containing a single randomly generated Random_Number.
2. WHEN a GET request is made to the GET_Endpoint, THE Random_Number_Service SHALL return an HTTP 200 status code.
3. THE Random_Number_Service SHALL generate a Random_Number as an integer between 1 and 100 inclusive on each request.
4. WHEN two consecutive GET requests are made to the GET_Endpoint, THE Random_Number_Service SHALL generate each Random_Number independently.
5. THE Random_Number_Service SHALL expose the GET_Endpoint under the `/api` path prefix.

### Requirement 2: Display Random Number

**User Story:** As a user, I want to see a randomly generated number displayed on the Frontend, so that I can view the output from the Random_Number_Service.

#### Acceptance Criteria

1. WHEN the Frontend loads, THE Frontend SHALL make a GET request to the GET_Endpoint and display the returned Random_Number.
2. THE Frontend SHALL display the Random_Number in a clearly visible area on the page.
3. WHILE the Frontend is waiting for a response from the Random_Number_Service, THE Frontend SHALL display a loading indicator.

### Requirement 3: Refresh Random Number

**User Story:** As a user, I want to click a Refresh_Button to get a new random number, so that I can request a fresh number from the Random_Number_Service without reloading the page.

#### Acceptance Criteria

1. THE Frontend SHALL display a Refresh_Button on the page.
2. WHEN the user clicks the Refresh_Button, THE Frontend SHALL make a GET request to the GET_Endpoint.
3. WHEN the Frontend receives a response from the Random_Number_Service after a refresh, THE Frontend SHALL update the displayed Random_Number with the new value.
4. WHILE the Frontend is waiting for a response after clicking the Refresh_Button, THE Frontend SHALL disable the Refresh_Button to prevent duplicate requests.

### Requirement 4: Error Handling

**User Story:** As a user, I want to see a meaningful message when the service is unavailable, so that I understand why no number is displayed.

#### Acceptance Criteria

1. IF the Random_Number_Service is unreachable, THEN THE Frontend SHALL display an error message indicating the service is unavailable.
2. IF the Random_Number_Service returns a non-200 status code, THEN THE Frontend SHALL display an error message with the status code.
3. IF an error occurs during number generation, THEN THE Random_Number_Service SHALL return an HTTP 500 status code with a JSON error message.

### Requirement 5: CORS Support

**User Story:** As a developer, I want the Random_Number_Service to support cross-origin requests, so that the Frontend can communicate with the backend from a different origin.

#### Acceptance Criteria

1. THE Random_Number_Service SHALL include CORS headers in responses to allow requests from the Frontend origin.
2. WHEN a preflight OPTIONS request is made, THE Random_Number_Service SHALL respond with appropriate CORS headers and an HTTP 200 status code.

### Requirement 6: Frontend Styling

**User Story:** As a user, I want the Frontend to have a clean, responsive layout, so that the application is usable on different screen sizes.

#### Acceptance Criteria

1. THE Frontend SHALL use Bootstrap for layout and styling.
2. THE Frontend SHALL render correctly on screen widths from 320px to 1920px.
3. THE Frontend SHALL center the Random_Number display and the Refresh_Button vertically and horizontally on the page.

### Requirement 7: Local Development with Docker

**User Story:** As a developer, I want to run the application locally using Docker and Docker_Compose, so that I can develop and test in a consistent containerized environment.

#### Acceptance Criteria

1. THE Application_Container SHALL be defined in a docker-compose.yml file at the project root.
2. WHEN a developer runs `docker-compose up`, THE Docker_Compose SHALL start both the Random_Number_Service and the Frontend as separate services.
3. WHEN the Docker_Compose services are running, THE Frontend SHALL be accessible on a configured local port.
4. WHEN the Docker_Compose services are running, THE Random_Number_Service SHALL be accessible on a configured local port.
5. THE Docker_Compose configuration SHALL use the Env_File to inject environment variables into the Application_Container.

### Requirement 8: Cloud Deployment on ECS

**User Story:** As a developer, I want to deploy the application to an ECS_Cluster in the us-east-2 region, so that the application is available in a production cloud environment.

#### Acceptance Criteria

1. THE application SHALL be deployable to an ECS_Cluster in the us-east-2 AWS region.
2. WHEN deployed to the ECS_Cluster, THE Random_Number_Service SHALL be running and reachable via a public or load-balanced endpoint.
3. WHEN deployed to the ECS_Cluster, THE Frontend SHALL be configured to communicate with the deployed Random_Number_Service endpoint.

### Requirement 9: Environment Configuration via Env File

**User Story:** As a developer, I want the application to read access tokens and configuration from an Env_File, so that sensitive credentials are managed outside of source code.

#### Acceptance Criteria

1. THE application SHALL read access tokens and configuration variables from the Env_File located at the project root.
2. THE Env_File SHALL NOT be committed to version control.
3. IF the Env_File is missing or unreadable, THEN THE application SHALL fail to start and log an error message indicating the missing Env_File.
4. WHEN the application starts, THE application SHALL validate that all required access tokens are present in the Env_File before accepting requests.

### Requirement 10: Local Python Environment Setup

**User Story:** As a developer, I want a requirements.txt file and a pre-configured virtual environment, so that I can install and manage Python dependencies consistently when running the Random_Number_Service locally.

#### Acceptance Criteria

1. THE project SHALL include a Requirements_File at the project root listing all Python package dependencies for the Random_Number_Service.
2. THE Requirements_File SHALL specify pinned versions for each dependency to ensure reproducible installations.
3. WHEN a developer runs `pip install -r requirements.txt` inside the Virtual_Environment, THE Virtual_Environment SHALL contain all packages needed to start the Random_Number_Service.
4. THE project SHALL use the existing Virtual_Environment located at `.venv` in the project root for local execution of the Random_Number_Service.
5. IF the Virtual_Environment directory does not exist, THEN THE application documentation SHALL instruct the developer to create the Virtual_Environment before running locally.

### Requirement 11: Infrastructure Code Organization

**User Story:** As a developer, I want all deployment infrastructure code kept in a dedicated `infra` directory at the project root, so that infrastructure concerns are clearly separated from application code.

#### Acceptance Criteria

1. THE project SHALL contain an Infra_Directory at the project root named `infra`.
2. THE Infra_Directory SHALL contain all Terraform configuration files required to provision and manage the cloud infrastructure.
3. THE Infra_Directory SHALL NOT contain application source code or runtime artifacts.

### Requirement 12: Self-Contained VPC and Networking Infrastructure

**User Story:** As a developer, I want the infrastructure to create its own VPC and all related networking resources, so that the deployment is fully self-contained and does not depend on pre-existing AWS networking resources.

#### Acceptance Criteria

1. WHEN the infrastructure is provisioned, THE Terraform configuration SHALL create a dedicated VPC for the application.
2. WHEN the infrastructure is provisioned, THE Terraform configuration SHALL create Public_Subnets within the VPC across at least two availability zones.
3. WHEN the infrastructure is provisioned, THE Terraform configuration SHALL create an Internet_Gateway and attach it to the VPC.
4. WHEN the infrastructure is provisioned, THE Terraform configuration SHALL create Route_Tables that route internet-bound traffic from Public_Subnets through the Internet_Gateway.
5. WHEN the infrastructure is provisioned, THE Terraform configuration SHALL create Security_Groups to control inbound and outbound traffic for the Application_Load_Balancer and ECS tasks.
6. THE infrastructure SHALL NOT depend on any pre-existing VPC, subnet, or networking resources in the AWS account.

### Requirement 13: Public Access to ECS Service

**User Story:** As a user, I want the deployed application to be publicly accessible from the internet, so that I can access the Frontend and Random_Number_Service without a VPN or private network connection.

#### Acceptance Criteria

1. WHEN the infrastructure is provisioned, THE Terraform configuration SHALL create an Application_Load_Balancer in the Public_Subnets.
2. THE Application_Load_Balancer SHALL accept inbound HTTP traffic on port 80 from any source IP address.
3. WHEN a request is received, THE Application_Load_Balancer SHALL route traffic to the ECS tasks running the application.
4. WHEN deployment is complete, THE application SHALL be reachable via the Application_Load_Balancer DNS name from the public internet.

### Requirement 14: Terraform-Based Infrastructure Management

**User Story:** As a developer, I want all infrastructure defined and managed using Terraform, so that infrastructure changes are version-controlled, repeatable, and auditable.

#### Acceptance Criteria

1. THE Infra_Directory SHALL contain Terraform configuration files (`.tf` files) that define all cloud resources.
2. WHEN a developer runs `terraform init` in the Infra_Directory, THE Terraform configuration SHALL initialize successfully with all required providers.
3. WHEN a developer runs `terraform plan` in the Infra_Directory, THE Terraform configuration SHALL produce a valid execution plan showing resources to be created.
4. WHEN a developer runs `terraform apply` in the Infra_Directory, THE Terraform configuration SHALL provision all required cloud resources including the VPC, networking, ECS_Cluster, and Application_Load_Balancer.
5. WHEN a developer runs `terraform destroy` in the Infra_Directory, THE Terraform configuration SHALL remove all provisioned cloud resources.
6. THE Terraform configuration SHALL use local state storage for the Terraform_State file.


### Requirement 15: GitHub Repository Creation

**User Story:** As a developer, I want a GitHub_Repository to be created matching the current project directory name, so that the project has a remote repository for version control and CI/CD integration.

#### Acceptance Criteria

1. WHEN the setup script is executed, THE application SHALL create a GitHub_Repository with a name matching the current project directory name.
2. THE application SHALL authenticate with the GitHub API using the Access_Token stored in the Env_File.
3. IF the GitHub_Repository already exists with the same name, THEN THE application SHALL skip creation and report that the repository already exists.
4. IF the Access_Token is missing or invalid, THEN THE application SHALL fail with an error message indicating the authentication failure.
5. WHEN the GitHub_Repository is created, THE application SHALL configure the local Git repository to use the new GitHub_Repository as its remote origin.

### Requirement 16: Build and Deploy Pipeline via GitHub Actions

**User Story:** As a developer, I want a Build_Pipeline and Deploy_Pipeline defined in GitHub_Actions, so that both the Frontend and Random_Number_Service containers are automatically built and deployed to their respective ECS tasks.

#### Acceptance Criteria

1. THE project SHALL contain GitHub_Actions workflow files in the `.github/workflows` directory.
2. WHEN code is pushed to the main branch, THE Build_Pipeline SHALL build Docker container images for both the Frontend and the Random_Number_Service.
3. WHEN the Build_Pipeline builds container images, THE Build_Pipeline SHALL push the images to ECR.
4. WHEN the Build_Pipeline completes successfully, THE Deploy_Pipeline SHALL deploy the built container images from ECR to the corresponding ECS tasks in the ECS_Cluster.
5. THE Build_Pipeline SHALL tag each container image with the Git commit SHA for traceability.
6. IF the Build_Pipeline fails for either container, THEN THE Deploy_Pipeline SHALL NOT execute and the workflow SHALL report the build failure.
7. THE GitHub_Actions workflows SHALL use secrets or environment variables for AWS credentials and shall NOT hardcode sensitive values.

### Requirement 17: Post-Deployment Health Verification

**User Story:** As a developer, I want the Deploy_Pipeline to verify that ECS tasks remain healthy after deployment, so that failed deployments are detected automatically and the pipeline reports a failure.

#### Acceptance Criteria

1. WHEN the Deploy_Pipeline deploys new container images, THE Deploy_Pipeline SHALL perform an ECS_Task_Health_Check to verify that the ECS tasks reach a running state.
2. WHEN the ECS_Task_Health_Check is performed, THE Deploy_Pipeline SHALL wait for a configurable timeout period for the ECS tasks to stabilize.
3. IF any ECS task fails to reach a healthy running state within the timeout period, THEN THE Deploy_Pipeline SHALL fail and report which task did not stabilize.
4. IF any ECS task starts but stops or crashes within the health check period, THEN THE Deploy_Pipeline SHALL fail and report the task failure.
5. WHEN all ECS tasks pass the ECS_Task_Health_Check, THE Deploy_Pipeline SHALL complete successfully and report the deployment as healthy.


### Requirement 18: Container Image Registry

**User Story:** As a developer, I want Docker container images stored in ECR, so that the ECS_Cluster can pull images from a managed, private registry within the same AWS account.

#### Acceptance Criteria

1. WHEN the infrastructure is provisioned, THE Terraform configuration SHALL create ECR repositories for the Frontend and the Random_Number_Service.
2. THE ECR repositories SHALL be in the same AWS region (us-east-2) as the ECS_Cluster.
3. WHEN the Build_Pipeline builds container images, THE Build_Pipeline SHALL authenticate with ECR and push the tagged images to the corresponding ECR repositories.
4. WHEN the ECS tasks start, THE ECS tasks SHALL pull container images from the ECR repositories.

### Requirement 19: Frontend Served via Nginx Container

**User Story:** As a developer, I want the Frontend served by an Nginx_Container deployed as an ECS task, so that static frontend assets are served efficiently in the cloud environment.

#### Acceptance Criteria

1. THE Frontend SHALL be packaged into a Docker image that uses Nginx as the web server.
2. WHEN deployed to the ECS_Cluster, THE Nginx_Container SHALL serve the static Frontend assets.
3. WHEN the Nginx_Container receives a request, THE Nginx_Container SHALL serve the Frontend files on port 80.

### Requirement 20: ECS Fargate Launch Type

**User Story:** As a developer, I want ECS tasks to run on Fargate, so that the deployment does not require provisioning or managing EC2 instances.

#### Acceptance Criteria

1. THE Terraform configuration SHALL configure the ECS_Cluster to use Fargate as the launch type for all ECS tasks.
2. THE ECS task definitions SHALL specify Fargate-compatible CPU and memory configurations.
3. THE ECS tasks SHALL run without any EC2 instances in the ECS_Cluster.

### Requirement 21: ALB Path-Based Routing

**User Story:** As a developer, I want the Application_Load_Balancer to route requests based on URL path, so that backend API requests and frontend requests are directed to the correct ECS tasks.

#### Acceptance Criteria

1. WHEN a request with a path matching `/api/*` is received, THE Application_Load_Balancer SHALL route the request to the Random_Number_Service ECS task.
2. WHEN a request with a path that does not match `/api/*` is received, THE Application_Load_Balancer SHALL route the request to the Nginx_Container ECS task serving the Frontend.
3. THE Terraform configuration SHALL define ALB_Routing_Rules for path-based routing between the Frontend and the Random_Number_Service.

### Requirement 22: Frontend Relative Path Communication

**User Story:** As a developer, I want the Frontend to use a relative URL path to communicate with the Random_Number_Service, so that the Frontend works behind the Application_Load_Balancer without hardcoded backend URLs.

#### Acceptance Criteria

1. THE Frontend SHALL use a relative path (e.g., `/api`) to make requests to the Random_Number_Service.
2. THE Frontend SHALL NOT contain hardcoded absolute URLs for the Random_Number_Service.
3. WHEN deployed behind the Application_Load_Balancer, THE Frontend SHALL rely on ALB_Routing_Rules to reach the Random_Number_Service via the relative path.

### Requirement 23: CloudWatch Logging for ECS Tasks

**User Story:** As a developer, I want ECS task logs sent to CloudWatch_Logs, so that I can view and troubleshoot application logs from a centralized logging service.

#### Acceptance Criteria

1. THE ECS task definitions SHALL configure the `awslogs` log driver to send container logs to CloudWatch_Logs.
2. WHEN the infrastructure is provisioned, THE Terraform configuration SHALL create CloudWatch_Logs log groups for the Frontend and the Random_Number_Service ECS tasks.
3. WHEN an ECS task writes to standard output or standard error, THE log output SHALL appear in the corresponding CloudWatch_Logs log group.
4. THE CloudWatch_Logs log groups SHALL be in the same AWS region (us-east-2) as the ECS_Cluster.
